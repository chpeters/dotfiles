# Quality

Prefer small, clear changes that are easy to review and validate.

## Expectations

- Keep behavior easy to understand from the code and docs.
- Prefer simple names and explicit data flow.
- Avoid duplicating helpers or patterns when an existing one fits.
- Handle errors deliberately instead of hiding failures.
- Keep tests close to the behavior they protect.
- Update plans and docs when the change affects how future work should be done.

## Repeated issues

When the same mistake appears more than once, capture the preferred pattern here or in the relevant plan.
